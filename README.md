# git-assist

Система поиска и анализа влияния изменений над кодовой базой, построенная по схеме
retrieval-augmented generation (RAG). Принимает на вход монорепозиторий, индексирует его
содержимое и отвечает на два класса вопросов: поиск по коду, документации и конфигурации
на естественном языке, и анализ влияния коммита — что затрагивается ниже по цепочке
зависимостей при изменении конкретного файла.

## Постановка задачи

Поиск по большому репозиторию через `grep` быстро упирается в то, что запрос пользователя
и текст кода описывают одно и то же разными словами: вопрос "где обрабатывается вход
пользователя при логине" не пересекается лексически с функцией `authenticate_request`.
Векторный поиск снимает эту проблему, сопоставляя запрос и код в пространстве эмбеддингов,
а не по совпадению токенов.

Вторая задача — импакт-анализ — устроена иначе и не сводится к поиску: нужно точно знать,
какие файлы зависят от изменённого, транзитивно. Для этого репозиторий индексируется не
только как набор текстов, но и как граф зависимостей между файлами и символами.

## Архитектура

Система построена вокруг понятия снапшота — неизменяемого построения индекса над
репозиторием на конкретном коммите. Каждая запись в любом сторе помечена
детерминированным `snapshot_id` (хешем от репозитория, ветки и коммита); чтения всегда
идут только по снапшоту со статусом `ACTIVE`, а построение новой версии индекса происходит
в фоне и переключается атомарно — так индекс никогда не отдаёт частично построенный
результат.

Хранение разделено на три роли, каждая — за отдельным протоколом в `packages/common`, с
одной реализацией на роль:

- **векторный стор** (Qdrant) — эмбеддинги чанков, поиск по косинусной близости в рамках
  снапшота;
- **графовый стор** (Neo4j) — рёбра зависимостей `src → dst` между файлами и символами;
- **стор метаданных** (Postgres) — жизненный цикл снапшотов, реестр документов и чанков.

Вызывающий код зависит только от протокола (`VectorStore`, `GraphStore`, `MetaStore`), а
не от конкретного класса — адаптер выбирается фабрикой (`make_vector_store`,
`make_graph_store`, `make_meta_store`) по URL из конфигурации. Это обычный приём
разделения интерфейса и реализации, а не специфика этого проекта — здесь он проведён
последовательно по всем трём хранилищам.

Путь запроса: текст пользователя → (опционально) расширение через HyDE → эмбеддинг →
векторный поиск по активному снапшоту → реранкинг кандидатов → сборка контекста →
генерация ответа с проверкой, что каждое утверждение подкреплено конкретными чанками
контекста. Путь импакт-анализа отдельный и короче: коммит → изменённые файлы → обратное
транзитивное замыкание по графу зависимостей → итоговый набор затронутых файлов.

### Модели

Все три модели — Qwen3, и все обслуживаются одним Ollama.

- **Эмбеддинг** — `Qwen/Qwen3-Embedding-4B`. Модель обучена по схеме Matryoshka (нативная
  размерность до ~2560), в системе используется усечённая размерность 1024 — единственный
  источник истины для размера векторного стора.
- **Реранкер** — `Qwen/Qwen3-Reranker-0.6B`. Специально выбрана младшая модель линейки,
  чтобы уложиться в бюджет по времени на пути запроса. Это decoder-only модель,
  оценивающая пару запрос/документ через logits токенов "yes"/"no" (sentence-transformers
  называет это модулем `LogitScore`); Ollama отдаёт нужные logprobs через `/api/generate`,
  score = softmax(yes, no).
- **Генерация** — `qwen3:4b` через `/api/chat` со структурированным JSON-выводом (Ollama
  валидирует ответ по JSON Schema на своей стороне). Ответ — набор утверждений со ссылками
  на id чанков, каждое из которых затем проверяется на то, что оно действительно
  подтверждается процитированным текстом — это отсекает утверждения, не имеющие опоры в
  контексте.

### API-поверхность

`apps/api` — четыре HTTP-операции (`/health`, `/search`, `/answer`, `/impact`) поверх
FastAPI; сам роутер устроен как чистая функция запрос → ответ без привязки к транспорту,
ASGI — тонкая обвязка вокруг неё. Это чистый бекенд без веб-клиента — обращаться к нему
нужно напрямую по HTTP (curl, скрипт, другой сервис).

`apps/worker` принимает push-события GitLab через вебхук, материализует изменённый коммит
и пересобирает снапшот: ingest → chunking (структурный, по границам tree-sitter для кода,
посекционно для документов) → эмбеддинг → индексация в векторный стор → построение графа
зависимостей (импорты плюс точные cross-references из SCIP-индексаторов, где они
доступны) → активация нового снапшота.

## Запуск

Полный стек (GitLab, Qdrant, Neo4j, Postgres, Ollama, API и worker) поднимается через
`docker compose up`; конфигурация — в `.env` (скопировать из `.env.example` и заполнить
реальные значения).

Локальная разработка без Docker требует переопределить `GIT_ASSIST_*_URL` в окружении на
адреса `localhost` — они закомментированы прямо в `.env.example`.
